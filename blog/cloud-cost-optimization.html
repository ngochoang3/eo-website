Cloud

Giảm 40% chi phí cloud mà không giảm hiệu năng

20 tháng 5, 2025·9 phút đọc·EO Technology

Cloud bill của bạn tăng đều mỗi tháng dù traffic không tăng tương ứng? Đây là vấn đề rất phổ biến — theo Flexera 2024 State of the Cloud Report, trung bình 30% chi phí cloud là lãng phí. Bài này trình bày 6 kỹ thuật cụ thể, áp dụng được ngay mà không cần refactor lớn.

Context: Các con số trong bài dựa trên thực tế tối ưu cho nhiều dự án từ $200–$3.000/tháng trên AWS, GCP, và một số trên DigitalOcean. Tỷ lệ tiết kiệm sẽ khác nhau tùy kiến trúc cụ thể.

1. Right-sizing: Giảm instance type về đúng nhu cầu thực

Đây thường là nguồn lãng phí lớn nhất. Nhiều dự án chọn instance lớn "cho an toàn" lúc ban đầu và không review lại. AWS Cost Explorer, GCP Recommender, và Azure Advisor đều có tính năng right-sizing recommendation — miễn phí.

Cách làm: Xem CPU utilization trung bình 2-4 tuần qua. Nếu CPU trung bình dưới 20% và peak dưới 40%, bạn đang dư thừa ít nhất 1 tier. Downsize instance type (ví dụ từ t3.medium → t3.small) hoặc đổi family (từ compute-optimized sang general purpose nếu không cần).

15–25%tiết kiệm

2. Spot/Preemptible Instances cho workload không time-critical

Spot Instances (AWS) hay Preemptible VMs (GCP) là unused capacity mà cloud provider bán với giá giảm 60-90% so với on-demand — nhưng có thể bị thu hồi với 2 phút báo trước.

Phù hợp với: Batch jobs, data processing, CI/CD pipeline runners, ML training, dev/test environments. Không phù hợp: Database instances, stateful services, production API servers cần uptime 100%.

Chiến lược hỗn hợp: Dùng 2-3 on-demand instances (minimum capacity) + 5-10 spot instances (scale-out). Auto Scaling Group của AWS hỗ trợ mixed instances policy tự động.

60–90%cho eligible workload

3. Reserved Instances / Committed Use — cam kết để tiết kiệm

Nếu bạn chắc chắn sẽ dùng một lượng tài nguyên nhất định trong 1-3 năm tới, Reserved Instances (AWS) hay Committed Use Discounts (GCP) tiết kiệm 30-60% so với on-demand.

Lưu ý: Chỉ commit sau khi đã right-size. Committing một instance quá lớn khóa bạn vào lãng phí trong 1 năm. AWS Compute Savings Plans linh hoạt hơn — áp dụng cho bất kỳ EC2 instance type nào trong region, không gắn cứng vào instance type cụ thể.

30–60%vs on-demand

4. CDN Offload — giảm data transfer và compute

Data transfer là chi phí ẩn lớn — AWS tính $0.09/GB egress ra internet. Nếu bạn serve nhiều static content (image, video, file download) thẳng từ EC2 hoặc GCS/S3, chi phí egress cộng dồn rất nhanh.

Giải pháp: Đặt Cloudflare (free tier phù hợp với hầu hết trường hợp) hoặc CloudFront trước origin. Cloudflare không tính tiền bandwidth giữa Cloudflare và origin, chỉ tính bandwidth từ origin đến Cloudflare. Trong thực tế, 70-90% static traffic được serve từ cache, giảm đáng kể cả egress cost lẫn compute cost.

5. Auto-scaling thay vì over-provisioning cố định

Nhiều hệ thống chạy constant capacity để xử lý peak traffic, nhưng off-peak có thể chỉ cần 20% capacity đó. Với auto-scaling:

  • Horizontal scaling: Thêm/bớt instance dựa trên CPU/request rate. Phù hợp stateless services.
  • Scheduled scaling: Nếu pattern traffic predictable (ví dụ thấp ban đêm), scale down theo lịch — đơn giản và hiệu quả hơn reactive scaling.
  • Serverless cho spiky workload: Lambda/Cloud Functions tính tiền theo invocation, không tính khi idle. Phù hợp API có traffic không đều.

6. Storage Tiering — không phải dữ liệu nào cũng cần hot storage

S3 Standard tốn $0.023/GB/tháng. S3 Glacier Instant Retrieval tốn $0.004/GB/tháng — rẻ hơn 6 lần, truy cập trong milliseconds. S3 Glacier Deep Archive tốn $0.00099/GB/tháng — rẻ hơn 23 lần, nhưng restore mất 12 giờ.

S3 Intelligent-Tiering tự động chuyển object giữa các tier dựa trên access pattern — không cần quản lý thủ công. Phí monitoring nhỏ nhưng thường worth it cho bucket lớn hơn 100GB.

Với log files, backup, và dữ liệu audit cũ hơn 90 ngày — Glacier Instant Retrieval gần như luôn đủ. Implement Lifecycle Policy để tự động chuyển.

Tổng kết: Thứ tự ưu tiên

  1. Right-sizing trước tiên — ít rủi ro, impact ngay, không cần code changes
  2. Storage tiering — set-and-forget với Lifecycle Policy
  3. CDN Offload — giảm egress và compute cùng lúc, thường cải thiện cả performance
  4. Reserved Instances — sau khi right-size và biết chắc usage pattern
  5. Spot Instances — cho batch/dev workload
  6. Auto-scaling — cần testing kỹ hơn, nhưng impact lớn nhất dài hạn

Bắt đầu từ đâu? Mở AWS Cost Explorer (hoặc GCP Cost Management) → xem top 5 service tốn tiền nhất → đọc recommendation của họ. 80% tiết kiệm thường đến từ 20% nguồn chi phí lớn nhất.
